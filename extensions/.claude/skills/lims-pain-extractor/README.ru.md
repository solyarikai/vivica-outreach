# lims-pain-extractor — корпус болей конкурентных LIMS

## Зачем

Базовый `email-sequence` в gtm-mcp умеет писать письма по 12 правилам. Но он не знает — и не может знать без внешних данных — на что конкретно жалуются Lab Director'ы по конкретному конкурентному вендору LIMS.

Для competitor-conquest sequence (типа «вы на LabWare, вот почему люди уходят») нужны реальные, недавние, проверяемые жалобы. Не выдуманные. Не перефразированные до неузнаваемости.

Этот скилл строит структурированный корпус болей, где каждая боль:
- ссылается на публичный URL отзыва
- датирована 2024 годом или позже
- классифицирована в одну из 8 стандартных тем
- содержит цитату ≤15 слов (правило копирайта)

## Когда запускать

- Когда добавляем нового конкурента (сейчас 8 в списке)
- Раз в квартал — обновить корпус, отзывы накапливаются
- Перед запуском новой competitor-conquest кампании
- Когда пользователь спрашивает «на что жалуются на <вендор>?»

**Не запускать**, если:
- Нужен позитивный маркетинговый контент (используйте сайт самого вендора)
- Нужно написать само письмо (это работа `email-sequence`, скилл только кормит его данными)
- Нужно определить, какую LIMS использует конкретная лаба (это работа `lims-detector`)

| 1 | G2 | Лучший сигнал — поле «What do you dislike» |
| 2 | Capterra | Перепроверка G2 |
| 3 | SourceForge | Голоса small/mid market |

| 5 | TrustRadius | Длинные отзывы, нюансы |
| 6 | Reddit | Нефильтрованная честность когда есть |

Полный список URL по вендорам — в `references/vendor_review_urls.yaml`.

## 8 стандартных тем болей

| Тема | Примеры |
|---|---|
| `ux` | Захламлённый интерфейс, много кликов, медленная навигация |
| `performance` | Краши, зависания, долгая загрузка |
| `integration` | Тяжело подключить приборы, хрупкий HL7/FHIR, vendor lock-in |
| `support` | Медленный отклик, платная поддержка, нет знаний |
| `reporting` | Негибкие отчёты, проблемы с BO/Crystal, нельзя кастомизировать |
| `migration` | Тяжёлый импорт данных, потеря данных при онбординге |
| `pricing` | Скрытые расходы, сюрпризы per-user, обязательный консалтинг |
| `compliance` | Дыры в audit trail, проблемы с e-sig, версионирование |

## Как структурирована боль в корпусе

```yaml
- vendor: labware
  theme: integration
  count: 14                  # сколько отзывов упомянуло это
  intensity: 4.2             # средняя серьёзность 1-5
  most_recent: 2025-09       # последний отзыв на эту тему
  quote: "Integration with other instrument software needed more efforts."
  quote_length_words: 9      # проверка копирайта — должно быть ≤15
  source_url: https://...
  source_site: g2
```

## Куда складываем

```
~/.gtm-mcp/projects/vivica/competitor_pains/
├── corpus.json              # все вендоры, все темы, machine-readable
├── refreshed_at.txt         # дата последнего обновления
└── per_vendor/
    ├── labware.md
    ├── labvantage.md
    ├── qbench.md
    ├── creliohealth.md
    ├── ligolab.md
    ├── clinisys.md
    ├── cloudlims.md
    └── lims_net.md          # он же JTO в чате с Петром
```

Markdown по каждому вендору также копируется в репо
(`reference/competitors/<vendor>.md`), чтобы планы и battle-cards
могли его читать как часть версионируемого репозитория.

## Pipeline

```
1. По каждому вендору из references/vendor_review_urls.yaml:
     По каждому URL:
       - web_fetch страницу (с пагинацией)
       - извлекаем: имя ревьюера, рейтинг, текст dislike, дату
       - дедуп по hash(имя + вендор)
       - фильтр: 2024+

2. По каждому вендору кластеризуем dislike-тексты по темам через LLM
   (одна жалоба может попасть в несколько тем)

3. Скоринг по (vendor, theme):
     count × intensity × recency_decay
     топ-3 темы на вендора

4. По каждой топ-теме выбираем ОДНУ цитату ≤15 слов
   Если естественная цитата длиннее — перефразируем целиком,
   но source URL остаётся для проверки

5. Записываем corpus.json + per-vendor markdown.
```

## Правило копирайта (строго)

Каждая цитата ОБЯЗАНА:
- быть ≤15 слов из одного отзыва
- иметь публичный source URL
- использоваться максимум один раз на источник (нельзя цеплять цитаты из одного отзыва)

Если жалобу нельзя выразить меньше чем 15 словами — **перефразируем целиком**, цитату убираем, source URL остаётся.

## Как `email-sequence` это использует

Когда `email-sequence` пишет competitor-conquest письмо для вендора X, он читает `corpus.json`, берёт топ-тему для X, вставляет цитату естественно:

> Subject: that LabWare integration headache  
> Hey {{firstName}},  
>
> Lab Director написал на G2 про LabWare: *"Integration with other instrument software needed more efforts."*  
> Большинство клинических лаб, перешедших на Vivica с LabWare, называют именно эту боль. Готовы за 10 минут показать, как мы решаем это для {{specialization}}-сегмента?

Цитата делает работу убеждения. Мы даём инструмент, не своё мнение.

## Частота обновления

- **Первый раз**: при инициализации проекта (сейчас)
- **Квартально**: обновляем, отзывы накапливаются
- **Ad-hoc**: добавили нового конкурента или у кампании упал reply rate (значит боли устарели)

## Запуск

```bash
# Полное обновление по всем 8 вендорам
python scripts/extract_pains.py \\
    --vendors-file references/vendor_review_urls.yaml \\
    --output ~/.gtm-mcp/projects/vivica/competitor_pains/ \\
    --repo-mirror ~/code/vivica-outreach/reference/competitors/

# Один вендор (например, только QBench)
python scripts/extract_pains.py \\
    --vendor qbench --output <dir> --repo-mirror <dir>
```

## Чего скилл НЕ делает

- НЕ меняет email-sequence или его 12 правил
- НЕ пишет письма (это работа email-sequence, скилл только кормит данные)
- НЕ детектит, какую LIMS использует лаба (это `lims-detector`)
- НЕ скрейпит позитивный контент или сравнения фич (только боли)
