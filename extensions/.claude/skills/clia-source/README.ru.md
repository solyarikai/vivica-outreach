# clia-source — POS-файл CMS как основной источник лидов

## Зачем этот скилл

Стандартный gtm-mcp ищет компании через Apollo. Для Vivica это не оптимально:

1. Клинические лабы — регулируемая отрасль, и **canonical-реестр это CMS QCOR**, а не crawl Apollo
2. Самый ценный триггер — «свежесертифицированная CLIA лаба» — **в Apollo не существует вообще**, он есть только в CMS-данных
3. У Apollo индустрия "Hospitals & Healthcare" слишком широкая, большинство — не лабы
4. Apollo берёт кредиты за каждую компанию, POS-файл бесплатный

Поэтому для Vivica путь такой:

```
POS-файл CMS (676 тыс строк)
   │
   │  этот скилл: фильтр + сегментация + форматирование
   ▼
ICP-компании (~9 тыс после фильтра)
   │
   │  стандартный gtm-mcp: скрейпит сайты
   ▼
Подтверждённые таргеты (~3-4 тыс после квалификации)
   │
   │  стандартный gtm-mcp: enrichment людей через Apollo
   ▼
Контакты с проверенными email
```

Этот скилл заменяет ТОЛЬКО первый шаг. Дальше всё — стандартный gtm-mcp.

## Что на входе

CSV из CMS, обычно называется `POS_File_CLIA_<quarter>.csv`. Скачивается с <https://qcor.cms.gov> через bulk export. В файле 100+ колонок, скилл использует около 15.

## ICP-фильтр — что считаем «таргетом Vivica»

Применяются ВСЕ условия:

1. **Активная** (`PGM_TRMNTN_CD == '00'`)
2. **Compliance или Accreditation** — лабы с Waiver не нуждаются в LIMS
3. **НЕ госпитальная** (`HOSP_LAB_EXCPTN_SW == 'N'`) — заперты в Epic/Cerner
4. **Есть телефон** — базовая проверка контактируемости
5. **Сертифицирована после 2025-01-01** (по умолчанию, можно поменять флагом `--since`)

На Q1 2026 файле с настройками по умолчанию выходит **~9 288 лаб**.

Можно сужать через `--since`:
- `2026-01-01` → ~4 800 (последние 4 месяца, самые горячие)
- `2025-06-01` → ~19 600 (последние 11 месяцев)
- `2025-01-01` → ~30 800 (без фильтра по дате)

## Сегментация

После фильтра каждая лаба тегается по специализациям из CLIA-кодов:

| Сегмент Vivica | CLIA-коды |
|---|---|
| `toxicology` | 800, 810 (токсикология, судебная токсикология) |
| `anatomic_pathology` | 100, 110, 120 (гистопатология, цитология) |
| `molecular_diagnostics` | 250 (молекулярка) |
| `clinical_chemistry` | 310, 320 (рутинная и автоматическая химия) |
| `hematology` | 400 (гематология) |
| `microbiology` | 500, 510, 520 (бактериология) |
| `immunology` | 600 |

Лаба может быть в нескольких сегментах одновременно.

POL/PSC/Reference определяется эвристиками (CMS этого напрямую не кодирует):
- **POL** (Physician Office Lab): `LAB_SITE_CNT == 1` И в названии есть "MD", "Dr ", "P.C." и т.п.
- **PSC** (Patient Service Center): в названии "lab", "diagnostic", "patient service", `LAB_SITE_CNT == 1`
- **REFERENCE**: `LAB_SITE_CNT > 1` ИЛИ объём тестов >50 000/год

## Русскоязычный сегмент (для Эндрю)

По заметке Эндрю с кик-офф звонка, русскоязычные владельцы — в основном NJ. Скилл генерирует **только список кандидатов** для ручной проверки:

```python
кандидаты = ICP-лабы из NJ/NY/PA
   которые имеют типичные паттерны имён клиник
```

Финальное решение «русскоязычный или нет» принимает Эндрю. Скилл не пытается это автоматизировать — слишком высокая цена ошибки (см. `playbooks/russian-segment.md` про различие подхода к русскоязычным vs обычным US-лабам).

## Что на выходе

Файл 1 — единый JSON со всеми отфильтрованными ICP-лабами:
`~/.gtm-mcp/projects/vivica/sources/clia_<quarter>.json`

Файл 2 — отдельные CSV по сегментам:
`~/.gtm-mcp/projects/vivica/sources/clia_<quarter>_segmented/`
- `toxicology.csv`
- `anatomic_pathology.csv`
- `molecular_diagnostics.csv`
- `russian_candidates_nj.csv`
- ...

Каждый CSV можно скармливать в `/launch` как отдельный сегмент.

## Что важно — у компаний нет доменов

В CMS-данных нет домена сайта лаборатории. Скилл выдаёт `domain: null`. Дальше отдельный шаг (Apollo `enrich_companies` по имени+адресу или Google-поиск) находит домены. Этот шаг **специально не делается внутри скилла** — он стоит Apollo-кредитов и должен быть подтверждён пользователем (правило gtm-mcp «никаких трат без подтверждения»).

## Как запустить

```bash
# Стандартный фильтр со всеми сегментами
python scripts/filter_pos_file.py \
  --input ~/code/vivica-outreach/input-data/POS_File_CLIA_Q1_2026.csv \
  --since 2025-01-01 \
  --output ~/.gtm-mcp/projects/vivica/sources/

# Только последние 4 месяца
python scripts/filter_pos_file.py \
  --input <file> --since 2026-01-01 --output <dir>

# Только токсикология
python scripts/filter_pos_file.py \
  --input <file> --segment toxicology --output <dir>
```

Скрипт печатает сводку и выходит. **API-вызовов нет. Кредитов не тратит.**

## Чего скилл НЕ делает

- НЕ обращается к Apollo (источник — POS-файл по дизайну)
- НЕ скрейпит сайты (дальше делает gtm-mcp scraping)
- НЕ enrich-ит людей (дальше делает gtm-mcp Apollo people enrichment)
- НЕ создаёт кампании (дальше делает gtm-mcp `campaign_push`)

Это чистый data-prep шаг, который встаёт в стандартный pipeline на месте «компании найдены».
